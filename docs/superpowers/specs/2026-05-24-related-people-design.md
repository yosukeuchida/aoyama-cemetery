# relatedPeople(主要な関係人物)機能 設計

- 作成日: 2026-05-24
- 目的: 各偉人ページに「青山霊園在住の主要な関係人物 3-7 名」を関係性ラベル付きで表示し、サイト内の偉人間クロスリンクを豊かにする
- 経緯: events データから自動算出する案を試したが、データの薄さで真の盟友が出ず断念。AI(Claude)による手動キュレーションに方向転換

## 動機

現状の偉人ページは「本文 → 作品 → 関連事件 → 地図」という構成で、人物間の関係性が「同事件への共起」でしか辿れない。これでは事件未登録の関係(師弟・血縁・派閥・後継・対立など)が表現できない。

具体例: 大久保利通の本来の重要関係人物は川路利良(警察制度の右腕)・伊地知正治(戊辰盟友)・松方正義(財政後継)・森有礼(教育改革仲間)・大木喬任(司法改革仲間)・牧野伸顕(次男)などだが、現状の events データ(50 件)から自動抽出すると黒田清隆・西郷糸子・後藤象二郎・副島種臣の 4 名しか出ず、しかも面子が「明治六年政変対立側」に偏ってしまう。

事件未登録の関係性を表現するには、AI による知識ベースのキュレーションが必要。

## スコープ

### 含む (v1)
- people コレクションの zod スキーマに `relatedPeople` フィールドを追加
- 全 90 偉人ページに関係人物 3-7 名 + 関係性ラベルをキュレーションして記入
- 偉人ページに「主要な関係人物」セクションを追加(カード型表示)
- カードに含む要素: 肖像サムネ + 氏名 + 役職 + 関係性ラベル + ページリンク

### 含まない (v2 以降)
- 関係人物の自動グラフ可視化(案 1、別フェーズ)
- 関係種別のタクソノミー化(まずは自由文で運用)
- relatedPeople[].slug の参照整合性検証スクリプト(zod では届かない範囲、必要なら push 前 sanity check として追加)
- 関係性の双方向自動 inverse(両側で個別にキュレーション)

## データ設計

### zod スキーマ追加

`src/content.config.ts` の people コレクションに optional フィールド追加:

```ts
relatedPeople: z.array(
  z.object({
    slug: z.string(),
    relation: z.string().min(1).max(60),
  })
).optional()
```

- `slug`: 関係先の人物 slug(`src/content/people/<slug>.md` に対応)
- `relation`: 関係性を本人視点で記述する自由文(例: 「警察制度の右腕として抜擢」「明治六年政変で対立」「次男」)。最大 60 文字
- `relatedPeople` 自体は optional(関係人物が見つからない場合は省略可能、まずは空のまま)

### frontmatter 記入例

`src/content/people/okubo-toshimichi.md`:

```yaml
relatedPeople:
  - slug: kawaji-toshiyoshi
    relation: 警察制度の右腕として抜擢した薩摩出身の腹心
  - slug: ijichi-masaharu
    relation: 戊辰戦争の盟友、薩摩藩参謀
  - slug: matsukata-masayoshi
    relation: 大蔵卿としての財政路線を継いだ後継者
  - slug: mori-arinori
    relation: 教育改革の同志、明六社にも参加
  - slug: ooki-takato
    relation: 司法・教育制度設計の同僚(佐賀)
  - slug: makino-nobuaki
    relation: 次男、後の内大臣
  - slug: soejima-taneomi
    relation: 明治六年政変で対立(佐賀)
```

### キュレーション原則

1. **対象は青山霊園在住の偉人のみ**(既存 90 名 = `src/content/people/*.md` に存在する slug)
2. **本人視点で関係を記述**(対象人物の側ではなく、当該ページの主人公から見た関係性)
3. **3-7 名** を目安(歴史的中心人物は 7 名、周辺人物は 3 名でも可)
4. **関係性の種類** は自由だが、以下のような視点を意識:
   - 師弟・育成: 「フルベッキに英語を学ぶ」「森有礼を文部行政に育てる」
   - 血縁: 「父」「次男」「娘婿」
   - 政治的盟友・対立: 「明治六年政変で対立」「終戦工作の盟友」
   - 派閥・所属: 「明六社の同人」「玄洋社の系譜」
   - 後継・前任: 「松方財政を継いだ後継者」「内務省を引き継いだ」
   - 同藩・同郷: 「薩摩・下加治屋町郷中の幼馴染」「佐賀の七賢人」
5. **fact-check は Wikipedia 本人記事冒頭 + 既存 `references`** を主ソース(既存事件リッチ化と同方針)

## 表示設計

### 配置

`src/pages/people/[slug].astro` の表示順:
1. 肖像 + 基本情報
2. 本文 markdown
3. 関連作品(works)
4. 関連事件(events)
5. **主要な関係人物(relatedPeople)** ← 新規
6. 地図(map)
7. 墓所写真ギャラリー

### UI

「関連事件」セクションと同じカード型レイアウトに統一:

```
## 主要な関係人物

[肖像]  川路 利良
        日本警察制度の父・初代大警視
        ↪ 警察制度の右腕として抜擢した薩摩出身の腹心

[肖像]  伊地知 正治
        戊辰戦争の薩摩藩参謀・伯爵
        ↪ 戊辰戦争の盟友、薩摩藩参謀

...(以下、3-7 名分)
```

- 肖像が無い偉人(PD 未経過 etc.)は名前イニシャル等のプレースホルダー
- カードクリックで対象人物ページへ遷移
- セクション全体は `relatedPeople` が空 / 未設定の場合は非表示

## 実装方針

### 工程

1. **設計レビュー** (本ドキュメント) → commit
2. **スキーマ + UI 実装**: `src/content.config.ts` 変更 + `src/pages/people/[slug].astro` にセクション追加。サンプル 1-2 名で動作確認
3. **キュレーション本番**: 6 並列 agent で 90 名 ÷ 6 = 15 名/グループに分担。本日の事件リッチ化と同パターン
4. **build 検証**: `npm run build` で zod schema が通ること(unknown slug 参照や relation 長すぎ等で fail しないこと)
5. **目視確認**: `npm run dev` で代表偉人ページ(大久保利通・犬養毅・西竹一など)の表示崩れチェック
6. **commit & push**

### 副作用・リスク

- **slug typo リスク**: zod では「文字列であること」しか検証できないため、存在しない slug 参照が発生する可能性。回避策として agent には「`/src/content/people/` に存在する slug のみを参照すること」を厳命。最終 build 後に簡易 sanity check script を 1 度走らせて参照整合性をチェック(v2 で常時化検討)
- **キュレーション偏り**: AI 知識の偏りで「教科書的に有名な関係」に偏る可能性。許容範囲とする(v2 で人間レビューにより補正)
- **bold 記法**: プロジェクトルール通り relation 文中に `**` は使わない
- **frontmatter コロン**: relation 文中にコロンを含む場合はダブルクオート必須(既存 CLAUDE.md ルール)

## 完了基準

- 全 90 偉人ページに `relatedPeople` 3-7 名がキュレーションされている
- `npm run build` が 0 エラーで通る
- 代表 5 ページの目視確認で関係性ラベルが意図通り表示される
- relatedPeople[].slug の参照整合性が全エントリで担保されている(sanity script 1 度走らせ)

## 将来展望(v2 以降)

本 v1 で蓄積される `relatedPeople` データは、案 1(関係グラフ)のデータソースに直接転用できる。relation の自由文を将来的にタグ化(`師弟` / `血縁` / `盟友` / `対立` 等)すればエッジ色で関係種別を可視化できる。

また各偉人で 5-7 名のキュレーションがあれば 90 × 6 = 540 エッジ程度になり、Cytoscape 等で 1 枚絵のネットワーク図として描画して「青山霊園 in-residence 関係ネットワーク」を提供できる。
